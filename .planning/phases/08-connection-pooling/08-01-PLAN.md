**Key decisions from CONTEXT.md (locked):**
- `Pop3ClientBuilder` is the account registration unit -- callers pass a builder with connection info and credentials already set
- Internal key derived from (host, port, username) extracted from the builder
- bb8's `PooledConnection<Pop3ConnectionManager>` returned on checkout -- standard RAII, auto-returns to pool on drop
- Busy checkout blocks with configurable timeout (bb8's `connection_timeout`)
- Pool gated behind a `pool` feature flag
- NOOP probe on checkout via bb8's `test_on_check_out(true)`
- Pool manages raw `Pop3Client` -- does NOT wrap with `ReconnectingClient`
- `Pop3ConnectionManager` implements bb8's `ManageConnection` trait: `connect()` builds + authenticates via stored builder, `is_valid()` sends NOOP, `has_broken()` checks `is_closed()`
- Granular error variants -- callers can distinguish checkout timeout, connection failure, authentication failure
- thiserror with `#[from]` / `#[source]` for source chaining

**From CONTEXT.md (Claude's Discretion):**
- Wrap bb8 types behind our own types (use newtypes for cleaner API)
- Implement `remove_account()` for deprovisioning
- Idle timeout and max lifetime defaults: use sensible defaults (no idle timeout, no max lifetime -- connections are cheap and POP3 servers have their own inactivity timeouts)
- Failed connection handling: bb8 auto-discard via `has_broken()` check
- Error type: new `Pop3PoolError` enum (aligned with feature flag boundary -- pool feature gets its own error type)

**From RESEARCH-concurrency.md (critical):**
- Use `tokio::sync::RwLock<HashMap>` (NOT DashMap) -- avoids new dependency, no async deadlock footgun
- Double-checked locking pattern: read-check (drop guard), build candidate with `build_unchecked()`, write-check (second check inside write lock, return winner)
- `build_unchecked()` is synchronous -- NEVER use `build().await` inside a lock
- `min_idle(None)` is mandatory -- connections created lazily on first checkout
- Return the canonical pool from the map, not the locally-built candidate

  ```rust
  //! Connection pool for managing multiple POP3 accounts concurrently.
  //!
  //! # RFC 1939 Exclusive-Lock Constraint
  //!
  //! **POP3 forbids concurrent access to the same mailbox.** Per RFC 1939 section 8,
  //! a POP3 server acquires an exclusive lock when a session enters the TRANSACTION
  //! state. A second connection to the same mailbox will be rejected with
  //! `-ERR maildrop already locked`. This pool enforces the constraint by capping
  //! each per-account pool at `max_size(1)` — at most one live connection exists
  //! per mailbox at any time.
  //!
  //! This pool is a **registry of per-account pools**, not a traditional N-connection
  //! pool to a single server. Each account (identified by host, port, and username)
  //! gets its own [`bb8::Pool`] with a single-connection limit. Different accounts
  //! can be checked out concurrently without blocking each other.
  //!
  //! # Usage
  //!
  //! ```no_run
  //! use pop3::{Pop3ClientBuilder, pool::{Pop3Pool, Pop3PoolConfig}};
  //!
  //! #[tokio::main]
  //! async fn main() -> Result<(), Box<dyn std::error::Error>> {
  //!     let pool = Pop3Pool::new(Pop3PoolConfig::default());
  //!
  //!     // Register an account
  //!     let builder = Pop3ClientBuilder::new("pop.example.com")
  //!         .credentials("alice", "secret");
  //!     pool.add_account(builder)?;
  //!
  //!     // Check out a connection — authenticated and health-checked
  //!     let mut conn = pool.get("pop.example.com", 110, "alice").await?;
  //!     let stat = conn.stat().await?;
  //!     println!("{} messages", stat.message_count);
  //!     // Connection returned to pool on drop
  //!     Ok(())
  //! }
  //! ```
  ```

     ```rust
     /// A connection pool for managing multiple POP3 accounts concurrently.
     ///
     /// # RFC 1939 Exclusive-Lock Constraint
     ///
     /// **POP3 forbids concurrent access to the same mailbox.** Each account in
     /// this pool is backed by a [`bb8::Pool`] with `max_size(1)`. A second
     /// checkout for the same account blocks until the first connection is
     /// returned. Different accounts can be checked out concurrently.
     ///
     /// See the [module-level documentation](self) for details.
     pub struct Pop3Pool {
         pools: tokio::sync::RwLock<
             std::collections::HashMap<AccountKey, std::sync::Arc<bb8::Pool<Pop3ConnectionManager>>>,

         config: Pop3PoolConfig,
     }

     - `pub fn new(config: Pop3PoolConfig) -> Self` -- creates empty registry

     - `pub fn add_account(&self, builder: Pop3ClientBuilder) -> std::result::Result<AccountKey, Pop3PoolError>` -- NOT async. Extracts key from builder using hostname(), effective_port(), username(). Returns `Pop3PoolError::NoCredentials` if username() is None. Creates `Pop3ConnectionManager`, builds `bb8::Pool` with `build_unchecked(manager)` (synchronous), inserts into the map under a **synchronous `std::sync::RwLock`** -- wait, actually the pool uses `tokio::sync::RwLock`. Since `add_account` is a deliberate registration step (not a hot path), use a blocking approach:

       Actually, `add_account` must be async because it needs to write to the `tokio::sync::RwLock`. Make it `pub async fn add_account(...)`.

       ```rust
       pub async fn add_account(
           &self,
           builder: Pop3ClientBuilder,
       ) -> std::result::Result<AccountKey, Pop3PoolError> {
           let username = builder.username()
               .ok_or_else(|| {
                   let key = AccountKey {
                       host: builder.hostname().to_string(),
                       port: builder.effective_port(),
                       username: String::new(),
                   };
                   Pop3PoolError::NoCredentials(key)
               })?
               .to_string();

           let key = AccountKey {
               host: builder.hostname().to_string(),
               port: builder.effective_port(),
               username,
           };

           let manager = Pop3ConnectionManager::new(builder);
           let pool = std::sync::Arc::new(
               bb8::Pool::builder()
                   .max_size(1)
                   .min_idle(None)
                   .retry_connection(false)
                   .test_on_check_out(self.config.test_on_check_out)
                   .connection_timeout(self.config.connection_timeout)
                   .build_unchecked(manager),
           );

           let mut guard = self.pools.write().await;
           guard.entry(key.clone()).or_insert(pool);
           Ok(key)
       }
       ```

     - `pub async fn get(&self, host: &str, port: u16, username: &str) -> std::result::Result<bb8::PooledConnection<'static, Pop3ConnectionManager>, Pop3PoolError>` -- Constructs an `AccountKey`, looks up the per-account pool in the registry, returns `AccountNotFound` if missing, calls `pool.get_owned().await` (use `get_owned` not `get` because the guard lifetime is too short):

       ```rust
       pub async fn get(
           &self,
           host: &str,
           port: u16,
           username: &str,
       ) -> std::result::Result<bb8::PooledConnection<'static, Pop3ConnectionManager>, Pop3PoolError> {
           let key = AccountKey {
               host: host.to_string(),
               port,
               username: username.to_string(),
           };

           let pool = {
               let guard = self.pools.read().await;
               guard.get(&key).map(std::sync::Arc::clone)
           };

           match pool {
               Some(p) => {
                   let conn = p.get_owned().await.map_err(Pop3PoolError::Pool)?;
                   Ok(conn)
               }
               None => Err(Pop3PoolError::AccountNotFound(key)),
           }
       }
       ```

       NOTE: Use `get_owned()` (returns `PooledConnection<'static, M>`) instead of `get()` (returns `PooledConnection<'_, M>`) because the RwLock guard is dropped before the bb8 checkout. The Arc keeps the pool alive. This is critical -- `get()` borrows from `Pool<M>` which would be dropped with the guard.

     - `pub async fn remove_account(&self, host: &str, port: u16, username: &str) -> bool` -- removes the account from the registry. Returns true if it was present. Any checked-out connections from the removed pool continue to work until dropped, but new checkouts for that key will fail with `AccountNotFound`.

       ```rust
       pub async fn remove_account(&self, host: &str, port: u16, username: &str) -> bool {
           let key = AccountKey {
               host: host.to_string(),
               port,
               username: username.to_string(),
           };
           let mut guard = self.pools.write().await;
           guard.remove(&key).is_some()
       }
       ```

     - `pub async fn accounts(&self) -> Vec<AccountKey>` -- returns a snapshot of all registered account keys.

     - `pub async fn pool_count(&self) -> usize` -- returns the number of registered accounts.

     - `pub async fn contains_account(&self, host: &str, port: u16, username: &str) -> bool` -- checks if an account is registered.
