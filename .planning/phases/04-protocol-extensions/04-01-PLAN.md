## Context

Phase 4 adds protocol extensions to the v2.0 POP3 client. This plan implements two tightly coupled features: RFC 2449/3206 RESP-CODES parsing and RFC 1939 APOP authentication. They share error.rs modifications and APOP's error handling depends on RESP-CODE variants existing.

### User Decisions (from CONTEXT.md -- NON-NEGOTIABLE)

- **4 new RESP-CODE variants**: `MailboxInUse`, `LoginDelay`, `SysTemp`, `SysPerm` added to `Pop3Error`
- **`[AUTH]` maps to `AuthFailed`**: no separate `AuthError` variant. The `[AUTH]` RESP-CODE merges into `AuthFailed`
- **Unknown codes fall through**: unrecognized RESP-CODEs fall to `ServerError(String)` with full text
- **`login()` updated**: must handle the case where `parse_status_line` now returns `AuthFailed` (from `[AUTH]`) instead of `ServerError`
- **APOP dual availability**: standalone `client.apop(user, pass)` method AND builder `.apop(user, pass)` (builder in Plan 02)
- **`#[deprecated]` attribute**: `apop()` method carries `#[deprecated(note = "...")]` with prominent `# Security Warning` rustdoc
- **No silent fallback**: if APOP requested but no timestamp in greeting, return `Pop3Error` immediately
- **Separate methods**: `.credentials()` for USER/PASS and `.apop()` for APOP (builder in Plan 02)

### Claude's Discretion (from CONTEXT.md)

- Whether RESP-CODE variants strip the bracketed code from the message string or preserve it -- **Decision: strip the bracket code, keep only the text after `]`** (consistent with `ServerError` stripping `-ERR`)
- UTF8 RESP-CODE inclusion -- **Decision: omit** (rarely seen in practice, no practical value)
- Exact `#[deprecated]` message wording -- use: `"APOP uses MD5 which is cryptographically broken. Prefer login() over a TLS connection."`

<task id="1" title="Add RESP-CODE error variants and update parse_status_line">

### What
Add 4 new `Pop3Error` variants for RESP-CODES, implement `parse_resp_code()` helper in `response.rs`, and update `parse_status_line()` to dispatch `-ERR [CODE]` responses to typed variants. Update `login()` in `client.rs` to also promote `AuthFailed` (from `[AUTH]`) in its error mapping.

### Why
CMD-04 requires server RESP-CODES to be parsed into structured `Pop3Error` variants. Currently all `-ERR` responses produce `ServerError(String)`. After this task, callers can `match` on `MailboxInUse`, `LoginDelay`, `SysTemp`, `SysPerm`, or `AuthFailed` (for `[AUTH]`).

### How

**Step 2: Add `parse_resp_code()` to `src/response.rs`**

Add this private helper function:

```rust
/// Map a `-ERR` response text with an optional RESP-CODE bracket prefix to a
/// typed `Pop3Error` variant. If no bracket code is found or the code is
/// unrecognized, falls through to `ServerError`.
fn parse_resp_code(text: &str) -> Pop3Error {
    if let Some(inner) = text.strip_prefix('[') {
        if let Some(end) = inner.find(']') {
            let code = &inner[..end];
            let rest = inner[end + 1..].trim_start().to_string();
            return match code {
                "IN-USE"      => Pop3Error::MailboxInUse(rest),
                "LOGIN-DELAY" => Pop3Error::LoginDelay(rest),
                "SYS/TEMP"    => Pop3Error::SysTemp(rest),
                "SYS/PERM"    => Pop3Error::SysPerm(rest),
                "AUTH"        => Pop3Error::AuthFailed(rest),
                _             => Pop3Error::ServerError(text.to_string()),
            };
        }
    }
    Pop3Error::ServerError(text.to_string())
}
```

**Step 4: Update `login()` error mapping in `src/client.rs`**

In `login()`, both the USER and PASS `.map_err` closures currently match only `Pop3Error::ServerError(msg) => Pop3Error::AuthFailed(msg)`. After RESP-CODES, a server returning `-ERR [AUTH] bad credentials` will produce `AuthFailed` directly from `parse_resp_code`. The `AuthFailed` variant will pass through the `other => other` arm unchanged, which is correct. However, we must also handle the case where the server returns `-ERR [IN-USE]` during login -- `MailboxInUse` should NOT be promoted to `AuthFailed`. Keep the existing match arms as-is; the current code is correct because:
- `ServerError(msg)` -> `AuthFailed(msg)` (plain `-ERR` without RESP-CODE)
- `AuthFailed(msg)` -> passes through `other => other` (already `AuthFailed` from `[AUTH]`)
- `MailboxInUse(msg)` -> passes through `other => other` (not an auth failure)

No changes needed to `login()` -- the existing `match` arms are already correct for RESP-CODES.

/// Extract the APOP timestamp from a server greeting.
///
/// The timestamp can appear anywhere in the greeting as `<...>`.
/// Returns `None` if no angle-bracket pair is found.
fn extract_apop_timestamp(greeting: &str) -> Option<&str> {
    let start = greeting.find('<')?;
    let end = greeting[start..].find('>')? + start;
    Some(&greeting[start..=end])
}

/// Compute the APOP MD5 digest: hex(md5(timestamp + password)).
fn compute_apop_digest(timestamp: &str, password: &str) -> String {
    let input = format!("{timestamp}{password}");
    let digest = md5::compute(input.as_bytes());
    format!("{:x}", digest)
}
```

**Step 3: Add `apop()` method to `impl Pop3Client`**

Add this method in the `impl Pop3Client` block, after `login()`:

```rust
/// Authenticate with the server using the APOP command (RFC 1939 section 7).
///
/// APOP uses an MD5 digest of the server greeting timestamp concatenated with
/// the password. The server must include a timestamp in its greeting
/// (e.g., `+OK POP3 server ready <1896.697170952@dbc.mtview.ca.us>`).
///
/// # Security Warning
///
/// **APOP uses MD5, which is cryptographically broken.** MD5 collision attacks
/// are practical and trivial. APOP provides no protection against offline
/// dictionary attacks if the exchanged messages are intercepted. Use only with
/// legacy servers where USER/PASS over TLS is unavailable. For modern servers,
/// prefer [`login()`](Self::login) over a TLS connection.
///
/// # Errors
///
/// - [`Pop3Error::ServerError`] -- server greeting has no APOP timestamp
/// - [`Pop3Error::AuthFailed`] -- server rejected the APOP credentials
/// - [`Pop3Error::NotAuthenticated`] -- called when not in Connected state
/// - [`Pop3Error::InvalidInput`] -- username or password contains CR or LF
///
/// # Example
///
/// ```no_run
/// use pop3::Pop3Client;
///
/// #[tokio::main]
/// async fn main() -> pop3::Result<()> {
///     let mut client = Pop3Client::connect_default("pop.example.com:110").await?;
///     // Server greeting must contain an APOP timestamp: <timestamp@host>
///     #[allow(deprecated)]
///     client.apop("user", "password").await?;
///     client.quit().await?;
///     Ok(())
/// }
/// ```
#[deprecated(note = "APOP uses MD5 which is cryptographically broken. Prefer login() over a TLS connection.")]
pub async fn apop(&mut self, username: &str, password: &str) -> Result<()> {
    if self.state != SessionState::Connected {
        return Err(Pop3Error::NotAuthenticated);
    }
    check_no_crlf(username)?;
    check_no_crlf(password)?;

    let timestamp = extract_apop_timestamp(&self.greeting)
        .ok_or_else(|| {
            Pop3Error::ServerError(
                "server does not support APOP (no timestamp in greeting)".to_string(),
            )
        })?
        .to_string();

    let digest = compute_apop_digest(&timestamp, password);
    let cmd = format!("APOP {username} {digest}");
    self.send_and_check(&cmd).await.map_err(|e| match e {
        Pop3Error::ServerError(msg) => Pop3Error::AuthFailed(msg),
        other => other,
    })?;

    self.state = SessionState::Authenticated;
    Ok(())
}
```
